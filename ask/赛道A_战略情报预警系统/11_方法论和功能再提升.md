### 企业场景与核心痛点
#### 企业场景
业务覆盖多个海外市场，面对竞争对手动态、产品趋势、社媒声量、平台策略及法规变动。市场资讯来源高度分散、更新频繁，需在不依赖内部数据的前提下，快速掌握各市场关键变化，以一致且可对比的方式汇总为决策参考。企业期望透过 AI Agent，模拟一个具备「市场雷达」能力的战略辅助角色，支持跨市场营运决策与日常策略判断。

#### 核心痛点
| 痛点 | 具体场景 |
| :--- | :--- |
| 市场信息高度分散 | 新闻、报告、社群资料分散，人工查阅耗时易遗漏 |
| 判断依赖个人经验 | 不同负责人标准不一，难以形成可对比结论 |
| 缺乏结构化决策输入 | 资讯零散，晨会中无法快速形成明确行动方向 |
| 市场变化与决策时间差 | 法规及平台政策调整往往事后才被注意到 |

---

### 赛题任务
#### 任务目标
构建海外市场战略情报 Agent，不接入任何企业内部数据，主动从多类型公开信息来源（新闻网站、社交媒体、行业报告、电商平台公告等）获取并整合市场动态，进行结构化分析与归纳，以「每日战略简报」输出，支持管理与运营团队快速判断与决策参考。

#### 输出要求示例
- 不同市场的关键变化要点（竞争 / 产品 / 平台 / 社媒 / 法规）
- 对业务潜在影响的判断（机会 / 风险 / 需关注事项）
- 建议的观察重点或后续行动方向

#### 开放性探索方向
- **信息筛选逻辑**：如何从海量公开信息中高效筛选有价值内容
- **分析框架设计**：结构化归纳的方法论，保证跨市场可对比性
- **输出结构与人机协同**：可追溯性设计及人工介入节点

## 一、现有架构与方法论是否最优？

### 1.1 结论：已高度匹配赛题，但有三块可以升级

目前的设计在“工程成熟度”和“分析框架完整性”上已经超出黑客松平均水平，但若追求**最优 + 落地真实企业**，有三处值得升级：

| 模块 | 当前方案 | 评价 | 升级建议 |
|------|---------|------|---------|
| 采集层 | Scrapy + 爬虫 | ★★★★ 工程化成熟 | 增加**事件驱动架构**（Webhook/SSE），减少轮询 |
| 分析层 | PEST + SWOT | ★★★★ 战略分析完善 | 引入**双重重要性评估**和**反身性假设**，防止Agent判断过于线性 |
| 应用架构 | Django 单体 | ★★★ 功能完整但紧耦合 | 引入 **Agent 协议**（A2A/MCP），解耦采集、分析、报告为独立智能体 |

## 二、升级方案：从“单体 Agent”到“多智能体协作架构”

### 2.1 为什么需要多智能体？

赛题强调“跨市场、多维度、人机协同”，真实企业落地时市场数量增加、信息量指数上升，单体 Agent 的提示词会越来越长，推理效率下降。更优的架构是**专业化分工**：

```
┌─────────────┐   ┌──────────────┐   ┌───────────────┐
│ 采集智能体   │ → │ 分类+归因智能体 │ → │ 简报撰写智能体  │
│ (数据获取)   │   │ (PEST/影响分析) │   │ (报告+可视化)   │
└─────────────┘   └──────────────┘   └───────────────┘
        │                │                    │
        └────────────┬───┘                    │
                     ↓                        ↓
              ┌─────────────┐        ┌──────────────┐
              │ 市场雷达总控  │ ←──── │ 人机协同终端   │
              └─────────────┘        └──────────────┘
```

**技术实现**：可采用 **Google A2A (Agent-to-Agent)** 或 **Anthropic MCP (Model Context Protocol)** 作为智能体间通信协议。这两个都是 2025–2026 年最新的 Agent 互操作标准，黑客松评审会视为技术领先加分项。

**工程落地**：保持 Django 作为主调度中心和 Web 面，但在内部引入 `AgentNode` 抽象类，每个节点是一个 Celery 任务，节点之间通过 Redis Streams 传递消息，遵循 A2A 语义。这样既有企业级稳定性，又具备智能体架构的灵活性。

### 2.2 分析框架升级：PEST+SWOT 之上再加两层

1. **双重重要性（Double Materiality）**  
   由欧盟 ESRS 提出，要求同时评估“事件对企业的财务影响”和“企业对外部环境的影响”。在你的场景中，可以要求 Agent 判断每个机会/威胁是否可能引发次生风险（如舆论反噬、合规连锁）。实现方式：在 SWOT 分析时增加一个反问 Prompt：“此事件若应对失当，会产生哪些二次影响？”

2. **反身性假设（Reflexivity）**  
   索罗斯理论：市场参与者的偏差会影响市场本身。适用场景：当监测到“多家媒体同时报道某个趋势”时，Agent 不仅报告趋势，还要发出“媒体报道本身可能加速该趋势”的提醒。这增加了分析的深度和独特性，评委眼中是高级策略思维的体现。

### 2.3 前端交互升级建议
目前设计的“驾驶舱 + 简报”已足够，但可增加一个 **“事件时间线”** 组件：  
将每个市场的关键政策、竞品动作、社媒节点按时间轴排布，一眼看清“某市场过去一周发生了哪些连锁变化”。这是企业晨会真实需求（“先看时间线，再看简报”），实现简单但感知价值极高。

## 三、面向真实企业落地的治理模型与工程检查清单

### 3.1 企业级治理模型（五个必须考虑的维度）

| 治理维度 | 核心问题 | 实现建议 |
|---------|---------|---------|
| **可观测性** | 爬虫挂了、LLM 超时、简报漏发怎么办？ | 接入 Sentry/OpenTelemetry；关键任务失败自动降级为仿真数据推送，保证简报不中断 |
| **数据血统** | 简报上某结论，怎么追溯回原始文章？ | 每条 RawInfo 带唯一 ID，简报中的每条结论用 `[ref: 原始ID]` 超链接，点击即到原文 |
| **多租户权限** | 不同市场负责人只能看自己市场的情报 | 数据库表增加 `tenant_id`，Django 中间件按用户角色过滤 |
| **合规与审计** | 爬虫是否遵守 robots.txt？数据留存多久？ | 爬虫中间件自动检查 robots.txt；数据库配置自动归档/删除策略 |
| **人工闭环** | 人工修正如何反馈给模型？ | 修正记录存入 `UserFeedback`，每周用这些案例微调 LLM 提示词（自动插入示例） |

### 3.2 全面检查清单（赛事后对照修正）

#### 一、信息源质量
- [ ] 信息源覆盖面：是否覆盖赛题要求的新闻/社交媒体/行业报告/电商公告/法规？
- [ ] 去重准确率：相同事件不同媒体来源，是否被合并？
- [ ] 数据新鲜度：时效性最高的信息是否在采集后 5 分钟内进入分析队列？
- [ ] 故障降级：是否有至少 3 个信息源出现故障时，仍能产出简报的熔断机制？

#### 二、分析准确性
- [ ] 维度分类准确率：人工抽检 100 条信息，PEST/五维分类正确率是否 ≥85%？
- [ ] 机会/威胁判断一致性：同一事件重复出现时，判断是否一致？
- [ ] 简报核心摘要：非专业人士阅读简报，是否能 30 秒内抓住要点？
- [ ] 误报率：标记为“高影响”但实际无战略意义的比例是否 <10%？

#### 三、系统性能与稳定性
- [ ] 数据写入性能：峰值 500 条/分钟 信息入库时不丢数据
- [ ] 简报生成延迟：从最新一条信息入库到简报渲染完成 <60 秒
- [ ] 爬虫可用性：核心 20 个信息源 24 小时可采集率 ≥95%
- [ ] 仿真数据切换：一键开关，爬虫故障后 30 秒内启动仿真模式

#### 四、可追溯性与透明性
- [ ] 每条简报结论有原文链接
- [ ] LLM 分析思维链可见（至少保留 3 步推理）
- [ ] 人工修正记录带时间戳和操作人
- [ ] 信息原始副本存档（即使源网站删稿）

#### 五、人机协同设计
- [ ] 用户可对简报结论进行确认/修正/忽略
- [ ] 被忽略的信息类型，一周内出现率下降（模型学习）
- [ ] 用户可自定义关注的市场、竞品、关键词
- [ ] 晨会模式：简报页面可一键投屏，进入演示模式（隐藏后台菜单）

#### 六、安全与合规
- [ ] 爬虫遵守目标站点 robots.txt
- [ ] API Key 等敏感信息不在前端或 Git 中暴露
- [ ] 用户登录有 session 超时机制
- [ ] 数据备份策略明确（每日自动备份）

#### 七、黑客松演示准备
- [ ] 准备 3 个市场 ×5 天 的仿真数据集，覆盖突发风险、机会出现、持续关注三类场景
- [ ] 录制 2 分钟核心流程演示视频（采集→分析→简报→通知）
- [ ] 准备评委可能的提问：如何处理信息冲突？如何评估 LLM 幻觉？如何控制成本？
- [ ] 凸显创新点：多智能体架构 + 双重重要性评估 + 事件时间线


## 四、最终建议：保持现有架构，有选择地升级

你现在的设计已经**超过了赛题的基准线**，不需要推倒重来。建议集中精力做三件事：

1. **引入多智能体协议概念**（哪怕只是一个 A2A 语义的 Celery 编排层），在 PPT 中重点展示“解耦协作”的架构先进性。
2. **实现事件时间线组件**，把简报中的信息串成因果关系图。
3. **完善降级与仿真切换**，确保任何演示环境下简报都能稳定产出。

以上这些升级完全基于你现有的 Django + Celery + Redis 架构，改动量可控，但能大幅提升项目在“落地真实企业”维度的得分。配合最后的检查清单，你的项目不仅是赛事作品，更是一套可直接向企业交付的战略情报系统原型。
