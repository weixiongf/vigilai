## 一、总体架构设计

```
┌──────────────────────────────────────────────┐
│                   前端展示层                    │
│     Django Templates / REST API / WebSocket   │
├──────────────────────────────────────────────┤
│                  Django 应用层                 │
│  用户管理 · 数据看板 · 报告生成 · LLM 调度     │
├──────────────┬──────────────┬────────────────┤
│   任务队列    │    缓存层     │    数据库层     │
│   Celery +   │    Redis     │  PostgreSQL    │
│  Redis 代理   │ 缓存 · 会话  │  数据持久化    │
└──────────────┴──────────────┴────────────────┘
        ▲              ▲              ▲
        │ 爬虫调度     │ 实时数据     │ 元数据管理
┌───────┴──────┐ ┌───┴────┐ ┌─────┴──────────┐
│  Scrapy 集群  │ │ WebSocket│ │  LLM 供应商 API│
│  + 爬虫状态表  │ │ 实时推送 │ │ (OpenAI等)     │
└──────────────┘ └────────┘ └──────────────┘
```

**分层职责**：
- **PostgreSQL**：唯一数据真相源，存储所有结构化数据
- **Redis**：高性能缓存、Celery 任务队列、WebSocket 通道、实时计数
- **Django**：业务逻辑中枢，管理 ORM、用户、权限、API、看板
- **Celery**：异步任务引擎，处理爬虫调度、数据分析、LLM 调用、报告生成、通知发送

```python
# apps/sources/models.py
class InfoSource(models.Model):
    """信息源注册表"""
    SOURCE_TYPE = [
        ('api', '免费API'),
        ('web', '网页爬取'),
        ('rss', 'RSS/Atom'),
        ('social', '社交媒体'),
    ]
    PRIORITY = [
        ('critical', '最高'),
        ('high', '高'),
        ('medium', '中'),
        ('low', '低'),
    ]
    
    name = models.CharField(max_length=200)            # 信息源名称
    url = models.URLField()                            # 目标地址
    source_type = models.CharField(max_length=20, choices=SOURCE_TYPE)
    priority = models.CharField(max_length=20, choices=PRIORITY)
    spider_name = models.CharField(max_length=100)      # 对应 Scrapy spider 名称
    crawl_frequency = models.IntegerField(help_text='爬取间隔(秒)')
    last_crawled_at = models.DateTimeField(null=True)
    last_status = models.CharField(max_length=20)       # success/error/disabled
    status_message = models.TextField(blank=True)
    is_active = models.BooleanField(default=True)
    
    class Meta:
        indexes = [models.Index(fields=['spider_name', 'is_active'])]
```

### 2. 原始信息存储

```python
class RawInfo(models.Model):
    """所有采集信息的主表（统一存储）"""
    source = models.ForeignKey(InfoSource, on_delete=models.CASCADE)
    title = models.CharField(max_length=500)
    content = models.TextField()                       # 原始内容
    summary = models.TextField(blank=True)             # LLM 生成摘要
    url = models.URLField(unique=True)                 # 去重关键字段
    published_at = models.DateTimeField()              # 信息发布时间
    fetched_at = models.DateTimeField(auto_now_add=True)
    language = models.CharField(max_length=10, default='en')
    raw_json = models.JSONField(null=True)             # 保留原始结构
    
    # 影响评估（LLM处理后填充）
    impact_score = models.IntegerField(null=True)      # 0-100 影响分数
    impact_direction = models.CharField(max_length=10, null=True) # positive/negative/neutral
    affected_metals = ArrayField(models.CharField(max_length=20), null=True) # ['gold','silver']
    sentiment = models.CharField(max_length=20, null=True) # bullish/bearish/neutral
    
    class Meta:
        indexes = [
            models.Index(fields=['source', 'published_at']),
            models.Index(fields=['impact_score']),
            models.Index(fields=['url']),              # 哈希索引用于去重
        ]
```

## 三、Redis 使用场景

| 场景 | Redis 数据结构 | 用途 |
|------|--------------|------|
| **Celery 消息队列** | Broker + Result Backend | 异步任务分发与结果存储（推荐 RabbitMQ 做 Broker 更可靠，但开发阶段 Redis 更简单） |
| **实时价格缓存** | String / Hash | 存储最新贵金属价格，Django 直接读取，无需查询 PostgreSQL |
| **爬虫状态锁** | String with TTL | 防止同一爬虫任务重复执行（`SET spider:gold_org:lock 1 EX 300 NX`） |
| **访问频率控制** | Sorted Set / String | 滑动窗口限流，防止爬虫被目标封禁 |
| **WebSocket 通道** | Pub/Sub | 新数据到达时实时推送到前端看板 |
| **趋势计数** | Sorted Set / HyperLogLog | 每小时/每天信息量统计，影响分数分布 |

**关键实现细节**：

### 1. 爬虫状态管理
- Django Admin 提供信息源启用/禁用、查看最后爬取状态
- 通过 Django REST Framework 暴露 API，Scrapy 的 `Spider Middleware` 在启动/完成时调用 API 更新 `CrawlJob` 状态
- Celery 定时任务 (`celery beat`) 根据 `InfoSource.crawl_frequency` 计算下次爬取时间，触发 Scrapy 爬虫执行

### 2. 数据存储流程
```
Scrapy Item Pipeline → HTTP POST 到 Django API → 
    URL 去重（Redis Bloom Filter 或 DB unique 约束）→ 
    写入 PostgreSQL RawInfo → 
    触发 Celery 任务：LLM 分析/影响评分
```

### 3. LLM 对接
- 抽象出 `BaseLLMProvider`，支持多个供应商（OpenAI / Claude / 本地 Llama）
- 提示词模板管理，可根据金属品种、语言定制
- 分析任务采用 Celery 异步执行，结果写入 `AnalysisReport`
- 设定重试机制和速率限制（通过 Redis）

### 4. 实时通知
- 高影响事件（impact_score > 阈值）触发 Celery 任务 → 调用企业微信/钉钉/飞书 Webhook
- 日报/周报定时生成，通过 Celery Beat 调度
- 失败重试与通知渠道健康检查

### 5. 数据展示
- 使用 Django Templates + Alpine.js/HTMX 构建轻量看板
- 更丰富的交互可使用 Django REST Framework + Vue/React 前端分离
- 实时数据通过 Django Channels (WebSocket) 推送，后端通过 Redis Pub/Sub 触发

```yaml
      - db
```

## 七、技术选型汇总

| 组件 | 推荐方案 | 替代选项 |
|------|---------|----------|
| Web 框架 | Django 4.2+ | FastAPI（若偏好异步） |
| 数据库 | PostgreSQL 16 | - |
| 缓存/队列 | Redis 7 | RabbitMQ（队列更可靠） |
| 任务队列 | Celery 5.3+ | Django-Q, Huey |
| WebSocket | Django Channels | - |
| REST API | Django REST Framework | Django Ninja |
| LLM 集成 | LangChain + OpenAI/Claude SDK | 直接 HTTP 请求 |
| 前端 | Django Templates + HTMX/Alpine | Vue 3 + Vite（后期可升级） |
| 部署 | Docker Compose | Kubernetes（生产环境） |
